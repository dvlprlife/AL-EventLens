# Change Log

All notable changes to the AL EventLens extension will be documented in this file. This project adheres to [Keep a Changelog](https://keepachangelog.com/en/1.1.0/) and uses [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [Unreleased]

### Changed

- CodeLens now caches the workspace-wide subscriber-count map per store generation instead of rebuilding it on every `provideCodeLenses` call (`src/ui/codelens.ts`). VS Code calls `provideCodeLenses` per visible `al` editor on every edit, scroll, and config change; each call recomputed the full publisher-key → subscriber-count map — O(total subscribers), tens of thousands at BaseApp scale. The map is now computed lazily and invalidated in `fireChange()`, the single point every count-changing signal (full re-index, incremental save, setting toggle) already routes through — mirroring the existing `EventTreeDataProvider` cache. The per-document `parseAl` stays per-call.
- Reading a `.alpackages/*.app` no longer copies the whole package payload before unzipping (`src/symbols/appReader.ts`). The 40-byte NAVX header is now skipped with a zero-copy `bytes.subarray(...)` view passed to JSZip instead of `bytes.slice(...)`, lowering peak memory on a large BaseApp. Also removed the unused `workspaceRelativePath` helper (`src/util/paths.ts`, now deleted — it had no callers).

### Fixed

- Event publishers declared inside `reportextension` objects in dependency `.app` packages were silently dropped from the index (`src/symbols/schemaFlat.ts`). `CONTAINER_KINDS` enumerated `Reports` but omitted the `ReportExtensions[]` array that `SymbolReference.json` uses for report-extension objects — the same gap class as the `TableExtensions`/`PageExtensions` omission fixed earlier — so a `[IntegrationEvent]` / `[BusinessEvent]` published from a compiled report extension was missing, even though the same publisher was indexed correctly from open workspace `.al` source (and any subscriber targeting it resolved as "unresolved"). The key is now enumerated alongside the base kinds; because the legacy flat parser and the BC 24+ nested `Namespaces[]` walk share this single enumeration, both schema surfaces are fixed.
- Event publishers declared inside `tableextension` / `pageextension` objects in dependency `.app` packages were silently dropped from the index (`src/symbols/schemaFlat.ts`). `CONTAINER_KINDS` enumerated the base object kinds but omitted the `TableExtensions[]` / `PageExtensions[]` arrays that `SymbolReference.json` uses for extension objects, so every `[IntegrationEvent]` / `[BusinessEvent]` published from a compiled extension object was missing — even though the same publisher was indexed correctly from open workspace `.al` source. The two keys are now enumerated alongside the base kinds, and because the legacy flat parser and the BC 24+ nested `Namespaces[]` walk share this single enumeration, both schema surfaces are fixed. (Verified against a real BC 28 Microsoft Base Application `SymbolReference.json`: the array keys are exactly `TableExtensions` / `PageExtensions`, and 109 `[IntegrationEvent]` publishers live inside extension objects there.)
- Bounded `.app` decompression to prevent a crafted dependency package from exhausting extension-host memory (`src/symbols/appReader.ts`). `readApp` materialized every `.app`'s `SymbolReference.json` and bundled `src/**/*.al` into JS strings with no size bound, so a few-KB zip-bomb `.app` in `.alpackages` whose entries declared multi-GB uncompressed sizes could OOM/hang the host. `parseAppBytes` now reads each entry's declared uncompressed size from the zip central directory (via JSZip's `_data.uncompressedSize`, read defensively) and rejects before inflating when a single entry exceeds 256 MB, the cumulative total exceeds 512 MB, or the bundled-entry count exceeds 50,000. The per-package `try/catch` in the indexer skips the one bad package and logs a clear "possible zip bomb" message; legitimate packages (including Microsoft BaseApp) are unaffected.
- A `.al` file save during an in-flight workspace-folder-change re-index no longer silently drops the rebuild (`src/index/folderWatcher.ts`). Adding a workspace folder kicks off a full re-scan; if the user saved a file before that scan finished, the save's generation bump suppressed the rebuild's commit (last-started-wins), so the newly-added folder's publishers/subscribers never appeared until a manual Refresh. The folder-change watcher now inspects the `committed` flag already returned by `runIndexAndCommit` and, when its rebuild was superseded, re-issues exactly one fresh full rebuild — guarded by a per-change sequence counter so a *newer* folder change (which commits on its own) cancels the stale re-issue and the re-issued run never arms a further one, leaving no possibility of a rebuild loop.
- UI/webview polish bundle (`src/ui/panel.ts`, `src/ui/subscriberTreeView.ts`, `src/ui/treeView.ts`). The Subscribers tree tooltip now shows a clean file path for subscribers parsed from a packaged `.app`'s bundled source instead of a backslash-mangled, scheme-stripped one — `Uri.fsPath` is used only for `file:` URIs; the synthetic `al-eventlens-app:` scheme falls back to `Uri.path` (matching the panel webview's existing guard). Export Mermaid invoked from the command palette after a Refresh no longer exports against a stale, dropped publisher — the cached panel selection is cleared on a full re-index. Re-opening the webview panel no longer leaks the previous disposed panel into the extension's disposable set, both activity-bar tree providers now dispose their `onDidChangeTreeData` emitter on shutdown (mirroring the CodeLens provider), and the webview CSP script nonce is now generated with `crypto.getRandomValues` rather than `Math.random()`.
- `stripComments` no longer treats `//` or `/*` inside an AL string literal (`'…'`) or quoted identifier (`"…"`) as a comment delimiter (`src/al/parser.ts`). The block-comment pass scanned the whole file with a regex, so a literal `/*` inside a string (e.g. `Message('progress: /* 50%')`) paired with a later genuine `*/` and silently blanked everything between — including any `[IntegrationEvent]` / `[BusinessEvent]` / `[EventSubscriber]` attribute in that span — dropping those events from the index with no error. The line-comment pass had the matching flaw: a string containing `//` (e.g. a URL) on an attribute line blanked the rest of the line, destroying the attribute's closing `)]` and dropping the subscriber. The two regexes are replaced with a single-pass character scanner tracking four exclusive states (single-quote string, double-quote quoted identifier, line comment, block comment), only starting a comment when in neither string state, and honoring AL's doubled-quote escapes (`''`, `""`). Byte offsets and newlines are preserved exactly, so all jump-to-source locations are unchanged. `src/al/triggers.ts` shares the same function, so commented-out Table/Page headers stay suppressed while real ones after a string containing comment delimiters are no longer dropped.
- `loadCachedSymbols` no longer throws on a corrupt cache entry, honoring its documented never-throws contract (`src/index/cache.ts`). The shape guard validated only the top-level wrapper (schemaVersion + three arrays), so a cache file that was valid JSON but had a malformed array element — a subscriber missing its `loc`, or a `{}`/`null` publisher (hand-edit or non-truncating disk corruption) — made the subscriber-revival `.map` throw a `TypeError` out of the function. A malformed subscriber dropped the whole app from the index until the `.app` mtime changed; a malformed publisher survived the loader and then threw in `resolveSubscribers` (outside the per-app try/catch), aborting the entire build. `loadCachedSymbols` now validates each publisher / subscriber / trigger-owner element after the top-level array check and returns `undefined` (a cache miss → clean re-parse) on any failure. The subscriber `loc` check requires non-negative integer `line` / `char` — `Number.isInteger(-1)` is `true` but `new vscode.Position(-1, 0)` throws `illegalArgument`. The flattened-`loc` revival is additionally wrapped in a `try/catch → undefined`, because `vscode.Uri.parse` throws `[UriError]` on a string whose scheme contains illegal characters (e.g. `"a b:c"`) even at the default `strict=false`, and validating that in the shape gate would force a `vscode` runtime call into the otherwise-pure validator — so the catch closes the whole revival throw class at the one site that touches the editor runtime. Hardening only — `storeCachedSymbols` always writes well-formed objects and the cache directory is not attacker-controlled.
- `findProcedureAfter` no longer scans the entire remaining file for the `procedure` keyword after each event attribute (`src/al/parser.ts`). For every `[IntegrationEvent]` / `[BusinessEvent]` / `[EventSubscriber]` match the parser sliced and ran the multiline `procedureRe` over the whole tail, so a file with many procedure-less attribute markers and a large tail was O(N × tail) — a crafted workspace `.al` (or a malicious `.app` bundling such a `src/*.al`) could freeze the extension host for seconds just by being opened, saved, or indexed (no progress, no abort). The keyword search is now bounded to a fixed window after the attribute (`PROCEDURE_SEARCH_WINDOW`, 2048 chars), capping per-match cost at O(window); the parameter list is still read against the full source so long multi-line signatures parse unchanged. Valid AL — where `procedure` always follows its decorator immediately — is unaffected.
- Under `alEventLens.includeAllAppVersions`, two physical copies of the **same** `(appId, version)` `.app` staged in different projects' `.alpackages` (e.g. the same dependency in a multi-root workspace) raced their cache-cleanup sweeps and evicted each other's freshly-written cache files, forcing an unnecessary full re-parse next session (`src/index/indexer.ts`). Pass 2 now collapses physically-identical same-`(appId, version)` packages before reading (`dedupByAppIdVersion`), so only one canonical copy enters the read/write pool. Genuinely distinct versions are unaffected — `includeAllAppVersions` still surfaces every version of an app.
- Trigger publishers for a Table/Page could be synthesized twice when its compiled `.app` twin was also present (`src/index/indexer.ts`, `src/al/triggers.ts`, `src/index/store.ts`). The trigger-owner dedup key lower-cased the object name but used the owning app's GUID verbatim, so an `app.json` `id` and a `NavxManifest.xml` `Id` differing only in case collated to different keys and the workspace-supersedes-package check missed — `triggerOwnerKey` / `collectTriggerOwners` now lower-case `appId` consistently. Separately, an incremental `.al` save re-applied the workspace triggers without evicting the bundled `.app` twin's set — they carry neither `location` nor `sourceUri`, so they slipped through `updateFile`'s survival filter — so a saved Table's `OnAfter*`/`OnBefore*` triggers appeared twice (inflating the publisher tree, CodeLens counts, and Mermaid edges) until the next full Refresh. `updateFile` now reapplies the workspace-supersedes-package rule on the incremental path, dropping any bundled trigger whose owner matches an incoming workspace trigger owner.
- Subscriber↔publisher matching no longer collides on quoted object or event names containing spaces (`src/index/match.ts`, `src/ui/panelHtml.ts`). `matchKey` and the webview's two hand-maintained copies (`keyOf` and the `rebuildSubscribersIndex` target key) joined `(kind, name, event)` with a single space, so `(name="A B", event="C")` and `(name="A", event="B C")` produced the same key — skewing resolved status and subscriber counts for space-containing names. All three now delimit with U+0001 (a C0 control char that can't appear in an AL identifier), matching the existing `subKey`.
- The rendered publisher signature no longer truncates or mis-splits a parameter list when a quoted parameter name or quoted type contains `(`, `)`, `[`, `]`, or `;` (`src/al/parser.ts`). `parseParameterListAt` and `splitParameterList` now skip over `"…"` quoted spans, so e.g. `Record "Weird ; Name"` and `"Quoted (Param)"` no longer break the paren-matching / semicolon-splitting scanners. Display-only — resolution was never affected.
- `alEventLens.gotoSubscriber` no longer surfaces a generic error toast on a malformed (negative or non-integer) range payload (`src/util/reviveLocation.ts`). `vscode.Position`/`Range` throw `illegalArgument` on a negative coordinate, and the command handler is not wrapped in `try/catch`; `revivePosition` now clamps negatives and floors non-integers to the documented `(0, 0)` fallback so jump-to-source always opens the file.

## [0.1.4] - 2026-05-24

### Added

- Subscribers section — a first-class view of every subscriber the workspace declares, **unresolved ones included** (`package.json`, `src/ui/subscriberTreeView.ts`, `src/ui/treeView.ts`, `src/ui/panel.ts`, `src/ui/panelHtml.ts`, `src/extension.ts`). Until now subscribers only appeared nested under a selected publisher, so a subscriber whose target app is missing from `.alpackages` — never matched to a publisher — was invisible in every surface. A second activity-bar tree (`alEventLensSubscribersView`, "Subscribers") now renders below "Publishers" and groups subscribers owning app → object kind → object → subscribed event, with a `✓` / `⚠` resolved badge per leaf. The webview panel gains a Publishers / Subscribers mode toggle; Subscribers mode lists every subscriber in a searchable flat list with a detail pane (owner, target, event, resolved status, jump-to-source). New command `alEventLens.revealSubscriber` and new `{type:'revealSubscriber', subscriber}` panel message — clicking a Subscribers-tree leaf opens the panel in Subscribers mode with that subscriber selected. `iconIdForKind` and `WORKSPACE_BUCKET` are now exported from `treeView.ts` so the subscriber tree reuses them rather than duplicating the icon map and workspace-bucket constant.

### Changed

- The webview panel's publisher and subscriber lists are now capped at 500 rendered rows (`src/ui/panelHtml.ts`). `renderList` / `renderSubscriberList` rebuilt the entire `<ul>` — one `<li>` per matching row — on every render, so on a large workspace (Microsoft BaseApp in `.alpackages` with trigger-event synthesis on, where the publisher list runs to tens of thousands of rows) the initial unfiltered render froze the panel. Both lists now stop appending past the cap and append a `Showing N of M — refine your search` notice row; the selected/revealed row is always rendered even past the cap so reveal and scroll-into-view still reach it. Searching narrows the list well below the cap.
- A `.al` file save now updates the webview panel incrementally — only the saved file's publishers and the re-resolved subscriber list are sent — instead of re-posting the entire index (`src/index/store.ts`, `src/ui/panel.ts`, `src/ui/panelHtml.ts`, `src/ui/treeView.ts`, `src/ui/subscriberTreeView.ts`, `src/ui/codelens.ts`). On a large workspace the index is tens of thousands of publishers, so re-cloning it across the webview boundary on every keystroke-save was needless work. The store now distinguishes a full-replace signal (`onDidChange`) from an incremental file-update signal (`onDidUpdateFile`).
- `buildIndex` now reads files with bounded concurrency instead of one at a time (`src/index/indexer.ts`, `src/util/concurrency.ts`). Workspace `.al` files, `.app` manifests, and `.app` packages were each read with sequential `await`, so a cold index of a large workspace was serialized I/O. Reads now overlap (up to 16 in flight) via a new `mapLimit` helper; parsing and index merging stay sequential and ordered, so the resulting index is unchanged.
- A warm symbol cache now skips re-reading a `.app` package entirely (`src/index/cache.ts`, `src/index/indexer.ts`). The cache stored only publishers, so `buildIndex` still fully decompressed every `.alpackages/*.app` on each index — to re-parse its bundled `src/**/*.al` for subscribers — even on a cache hit. The cache (schema v5) now also stores the bundled-source subscribers and trigger owners, so a hit reuses them and skips the `readApp` NAVX + PKZIP decompression. Subscriber resolution is still recomputed globally; pre-v5 cache files are re-parsed once on upgrade.
- The panel's subscriber detail pane now looks the selected subscriber up through a row-keyed index rather than a linear scan (`src/ui/panelHtml.ts`), so selecting a row stays instant on a workspace with many subscribers.

### Fixed

- Workspace trigger publishers accumulated a duplicate set on every `.al` save (`src/al/triggers.ts`, `src/index/indexer.ts`). `buildIndex` synthesized Table/Page trigger publishers without a `sourceUri`, so the `EventIndexStore` save-survival filter (`location.uri ?? sourceUri`) never matched them — the watcher's freshly-tagged set was appended each save instead of replacing the previous one, so CodeLens subscriber counts inflated, the panel listed each trigger twice (then thrice, then…), and Mermaid export emitted duplicate edges. Only a full Refresh recovered. `synthesizeTriggerPublishers` now accepts an optional `sourceUri`; the indexer's workspace pass pairs each trigger owner with the URI of the declaring `.al` file and threads it through synthesis so the resulting publishers carry the same `sourceUri` the watcher does, and the survival filter evicts the previous set cleanly. `.app`-bundled trigger owners deliberately keep `sourceUri: undefined` (they have no workspace source file; the `undefined` is what protects them from being evicted by a workspace-side save).
- Panel UX bugs around the new Subscribers mode (`src/ui/panelHtml.ts`). The subscriber detail pane no longer goes blank after a line-shifting save — the previously-selected subscriber is relocated by line-insensitive identity (owner, target, event, path), and only clears if zero or multiple matches exist. An externally-driven `select` of a publisher past the 500-row list cap now renders the list first so the row is highlighted and scrolled into view (matching how the existing `reveal` handler already worked). The "Showing X of Y — refine your search" notice no longer reports `MAX_LIST_ROWS + 1` when the selected row is past the cap. Changing the App dropdown now clears a stale `objectIdentity` selector the same way changing the Kind dropdown already did, so picking a different app from a tree-revealed `codeunit::"Sales-Post"` filter doesn't leave the list empty with no visible reason.
- Concurrent index runs and rapid same-file saves no longer overwrite the store with a stale snapshot (`src/index/reindex.ts`, `src/extension.ts`, `src/index/folderWatcher.ts`, `src/index/watcher.ts`). The activation pass, the `alEventLens.refresh` command, and the workspace-folder-change re-index now route through a new `runIndexAndCommit` helper that reserves a monotonic generation token at start and drops the late-resolving result if a newer run has taken over — so a slow initial pass cannot revert the store to its older state after a user-triggered Refresh has already committed a newer snapshot. `handleSave` carries a per-URI `Symbol()` token across the `discoverWorkspaceApps` await so two rapid saves of the same file no longer race; saves to different URIs still index in parallel. A successful `handleSave` bumps the started-generation counter so an in-flight rebuild can't overwrite the saved-file delta on commit. The activation-time empty-index failure fallback is gated on `!store.isInitialized` so a save that commits during a slow failing initial pass survives the failure handler.
- Indexer hardening (`src/index/indexer.ts`, `src/util/concurrency.ts`). Pass 1 now parses inside read workers rather than holding every workspace `.al` file's decoded text at once, capping peak memory on a BaseApp-scale workspace. Per-file read failures are isolated via a new `{ onError: 'skip' }` mode on `mapLimit`, so one transient `EBUSY`/`ENOENT` no longer aborts the whole build. Parser exceptions (as opposed to read failures) are kept loud: `parseAl` / `collectTriggerOwners` throws are wrapped with an `[AL EventLens parser bug]` prefix, surfaced to the user via `vscode.window.showErrorMessage` from `extension.ts`'s catch, and re-thrown so the failure isn't silently absent from the index. Pass 2 now reuses the pre-built `metaByUri` instead of re-reading every `.app` manifest a second time, and the Pass-2 merge preserves `isWorkspaceApp: true` on existing entries so a workspace `.app` that leaks past exclusion keeps the tree's workspace-first sort and `root-folder` icon.
- Cache hygiene (`src/index/cache.ts`, `src/index/indexer.ts`). Cache cleanup is now version-aware — concurrent writes for different versions of the same appId no longer evict each other — and `pruneOrphanCacheEntries` sweeps only files whose payload parses with a non-current `schemaVersion`. Previously the classifier treated any `(appId, version)` not in this run's `visitedKeys` as orphan, which wiped cache entries owned by *other* workspaces (since `globalStorageUri/symbols/` is shared) and also deleted perfectly valid current-schema files when `readFile`/`JSON.parse` threw transiently during the sweep. Module-scoped generation state is also reset from a new `deactivate()` hook so a Developer: Reload Window starts each activation from a clean counter.
- Publishers and Subscribers trees no longer split a single AL object across casing variants — the `groupByObject` bucket key is lowercased so `codeunit "MyCu"` and `codeunit MyCu` merge into one row (display label keeps the first-seen casing) (`src/ui/treeView.ts`, `src/ui/subscriberTreeView.ts`). The Subscribers tree refresh test now exercises `registerSubscriberTreeView` and asserts on both the `store.onDidChange` (full re-index) and `store.onDidUpdateFile` (incremental `.al` save) wires — a regression that dropped the save wire would previously have gone undetected (`src/test/suite/subscriberTreeView.test.ts`).

## [0.1.3] - 2026-05-20

### Fixed

- An app open in the workspace as `.al` source was also indexed a second time from its compiled `.app` in `.alpackages`, doubling every one of its publishers, subscribers, and synthesized trigger owners — subscriber-count badges and CodeLens counts were inflated for any app present on both the workspace and the package axis (`src/index/indexer.ts`, `src/index/appJson.ts`, `src/al/parser.ts`). This is distinct from the multi-version dedupe in 0.1.2, which deduped `.app` files *within* `.alpackages`; this bug was on the workspace ⇄ package axis. `buildIndex` now discovers every workspace project's `app.json` up front and, before version selection, drops any `.alpackages/*.app` whose manifest `appId` matches an open workspace project. Workspace `.al` source is authoritative — it carries `[EventSubscriber]` data that `SymbolReference.json` strips at compile time (per CLAUDE.md) — so skipping the compiled twin loses nothing. The skip applies regardless of `alEventLens.includeAllAppVersions` (all versions of a workspace app are suppressed) and the `appId` comparison is case-insensitive since GUID casing varies between `app.json` and `NavxManifest.xml`. A dependency-only `.app` and a workspace-only project are both unchanged.
- In a multi-root workspace, every AL project's publishers and subscribers were flattened into one undifferentiated `(workspace)` tree node because Pass 1 of `buildIndex` created them all with `owner.appId === undefined` (`src/index/indexer.ts`, `src/index/appJson.ts`, `src/al/parser.ts`, `src/index/watcher.ts`, `src/ui/treeView.ts`). The indexer now attributes each workspace `.al` file to the nearest enclosing `app.json` project (longest-path-prefix wins, so nested projects resolve to the innermost) and stamps that project's GUID onto the publishers, subscribers, and synthesized trigger owners it produces; each project's `app.json` `name` / `publisher` is contributed to `EventIndex.appMeta`. The activity-bar tree now shows each workspace project as its own top-level node — labeled from `app.json`, marked with the `root-folder` codicon, sorted before dependency packages (case-insensitive alphabetical within each group). The literal `(workspace)` node survives only as the fallback bucket for loose `.al` files under no `app.json`; a single-root workspace shows its project name instead. The incremental save watcher applies the same attribution so a saved file keeps its project. Subscriber resolution is unchanged — `resolveSubscribers` keys only on the subscriber's `target`, never its `owner.appId`.

## [0.1.2] - 2026-05-19

### Fixed

- Duplicate publishers when `.alpackages` carried multiple versions of the same app (`src/index/indexer.ts`, `src/symbols/appReader.ts`, `src/util/versions.ts`). A user with stock BC 26.5 + 27.0 + 28.0 BaseApp `.app` files in `.alpackages` was getting all three versions' events in the index — ~70k BaseApp events under one bucket instead of ~23k, with subscriber-count badges, CodeLens counts, and panel rows all multiplied. The indexer now groups `.alpackages/*.app` files by their manifest `appId` and keeps only the highest-`Version` URI per group (matching how the official AL extension resolves dependencies). New `readAppMetadata` / `parseAppMetadataBytes` exports do a cheap `NavxManifest.xml`-only read so losers don't pay the `SymbolReference.json` decompression or bundled-`src/**/*.al` walk. Ties on identical `(appId, Version)` resolve deterministically (first URI in `toString` order) with a `console.warn`. The previous behavior is preserved as an opt-in via the new `alEventLens.includeAllAppVersions` setting (default `false`), for multi-target / platform-version testing setups. Per-`.app` metadata-read failures during the dedupe pass tolerate the same `console.warn` + continue pattern the full-parse loop already uses.

### Added

- `alEventLens.includeAllAppVersions` setting (boolean, default `false`) — see the Fixed entry above. When `true`, restores the pre-fix behavior of indexing every `.app` file regardless of `appId` duplication.

## [0.1.1] - 2026-05-19

### Added

- Object click in the activity-bar tree now drives the panel filter (`src/ui/treeView.ts`, `src/ui/panel.ts`, `src/ui/panelHtml.ts`, `src/extension.ts`, `package.json`). Clicking an `ObjectNode` row reveals the panel and applies an `app:<id> kind:<k> object:<name>` filter so only that object's events show — the chevron still toggles expand/collapse independently. Clicking an `EventNode` leaf (or an inline CodeLens) applies the same filter and additionally selects the event inside it, so the revealed publisher is in view rather than buried in an unfiltered list. The panel's search box gains an `object:` prefix matching the existing `app:`/`kind:` pattern, plus quoted-token support (`object:"Sales Header"`) so object names with spaces round-trip cleanly. New command `alEventLens.revealObject` and new `{type:'reveal', search, selectKey?}` panel message.
- Event signature line in the panel detail pane (`src/ui/panelHtml.ts`, `src/al/parser.ts`, `src/symbols/schemaFlat.ts`) — when a publisher is selected, the right pane now shows the procedure's parameter list (e.g. `(var SalesHeader: Record "Sales Header"; CommitIsSuppressed: Boolean)`) right under the header, in monospace and themed with the description-foreground color. `var` is rendered as a separate themed span. Parameters are extracted from AL source for workspace publishers and from `SymbolReference.json` `Parameters[]` for `.alpackages/*.app` publishers — including length-bound types (`Code[20]`), subtyped records (`Record "Sales Header"`), and generic types (`List of [Code[20]]`). Synthesized trigger publishers carry no parameters and render no signature line. The cache schema bumps to v3 to invalidate v2 entries on the next session so they get re-parsed with parameters.
- AL source parser (`src/al/parser.ts`) recognizes publisher (`[IntegrationEvent]`, `[BusinessEvent]`) and subscriber (`[EventSubscriber]`, both pre-BC22 and BC22+ syntaxes) declarations across all AL object kinds, returning typed `Publisher` and `Subscriber` records with `vscode.Location` pointing at the procedure name.
- Trigger-event synthesis (`src/al/triggers.ts`) produces 10 virtual publishers per Table object (`OnBeforeInsertEvent`, `OnAfterInsertEvent`, `OnBeforeModifyEvent`, `OnAfterModifyEvent`, `OnBeforeDeleteEvent`, `OnAfterDeleteEvent`, `OnBeforeRenameEvent`, `OnAfterRenameEvent`, `OnBeforeValidateEvent`, `OnAfterValidateEvent`) and 8 per Page object (`OnOpenPageEvent`, `OnClosePageEvent`, `OnQueryClosePageEvent`, `OnInsertRecordEvent`, `OnModifyRecordEvent`, `OnDeleteRecordEvent`, `OnNewRecordEvent`, `OnAfterGetCurrRecordEvent`). The indexer will gate this on the `alEventLens.includeTriggerEvents` setting.
- `.app` package reader (`src/symbols/appReader.ts`) reads BC application packages via `vscode.workspace.fs` (web-compatible), validates the 40-byte NAVX header, decompresses the embedded PKZIP with JSZip, and extracts the `appId`, `version`, `SymbolReference.json` body, and bundled AL source files under `src/**`. `.NEA` runtime packages are detected by extension and rejected with a clear error rather than a silent failure.
- Legacy `SymbolReference.json` schema parser (`src/symbols/schemaFlat.ts`) extracts publisher events (`IntegrationEvent`, `BusinessEvent`) from the pre-BC24 flat schema for Codeunits, Tables, Pages, Reports, Queries, XmlPorts, and Interfaces. Accepts either a JSON string (with optional UTF-8 BOM, common in Microsoft `.app` packages) or an already-parsed object.
- Nested-namespaces `SymbolReference.json` schema parser (`src/symbols/schemaNamespaces.ts`) handles the BC24+ schema where object-kind arrays live recursively inside `Namespaces[]` entries. Walks both root-level arrays and nested namespaces at any depth. Reuses `extractPublishersFromContainer` from `schemaFlat.ts` so the per-object/per-method walk isn't duplicated. With this, `detect.ts` routes both schemas correctly: `Namespaces` key present → nested parser, otherwise → flat parser. The full `.app` ingestion pipeline (`readApp` → `parseSymbolReference`) is now operational; the indexer just needs to wire it into a workspace scan.
- Subscriber-to-publisher resolver (`src/index/resolver.ts`) sets the `resolved` flag on each subscriber by matching `(kind, name, eventName)` case-insensitively against the workspace publisher set. Trigger publishers participate equally with `[IntegrationEvent]` / `[BusinessEvent]` publishers; `Subscriber.target.appId` is ignored so cross-app resolution works. Pure function over already-parsed `Publisher[]` and `Subscriber[]` — no I/O, no `vscode` runtime imports — ready to compose into the workspace indexer.
- Workspace event indexer (`src/index/indexer.ts`) composes the AL parser, trigger synthesizer, `.app` reader, schema parsers, and subscriber resolver into a single `buildIndex(context)` call. Walks `**/*.al` under workspace folders and `**/.alpackages/*.app`, gates package scanning on `alEventLens.scanAlpackages` and trigger synthesis on `alEventLens.includeTriggerEvents`, tolerates a corrupted or `.NEA`-rejected `.app` per package via `console.warn` without aborting the run, and returns a fully resolved `EventIndex`. `extension.ts` invokes it during `activate()` and logs publisher/subscriber counts for verification — no panel, tree, or CodeLens wiring yet.
- Persistent `.app` symbol cache (`src/index/cache.ts`) stores parsed publisher lists under `extensionContext.globalStorageUri/symbols/{appId}__{version}__{mtime}.json` via `vscode.workspace.fs`, gated by `alEventLens.cache.enabled`. The indexer checks the cache before re-parsing each `SymbolReference.json` and cleans up older entries for the same `appId` on version bumps, making second-and-later workspace opens near-instant for the `.app` ingestion path.
- Long-lived event index store (`src/index/store.ts`) — the `EventIndexStore` class wraps the current `EventIndex` behind a `vscode.EventEmitter`-backed `onDidChange`, with `set` (full replace, used by the activation-time `buildIndex`) and `updateFile` (per-URI replace that re-runs `resolveSubscribers` so cross-file links stay correct). Trigger publishers carry an optional `sourceUri` so the store can identify and replace them on a subsequent save instead of accumulating duplicates. Pushed into `context.subscriptions` from `extension.ts` so disposal is automatic on extension shutdown.
- Save watcher (`src/index/watcher.ts`) — `registerSaveWatcher(context, store)` subscribes to `vscode.workspace.onDidSaveTextDocument`, filters AL documents, re-parses with `parseAl`, optionally synthesizes Table/Page trigger publishers (gated by `alEventLens.includeTriggerEvents`), and routes the result into `EventIndexStore.updateFile`. Gated by `alEventLens.indexOnSave`. The inner `handleSave` helper is exported so tests can drive it directly without stubbing the workspace event surface. The trigger-owner extraction helper (`collectTriggerOwners`) moved from `src/index/indexer.ts` to `src/al/triggers.ts` so both the full-pass indexer and the incremental save watcher can share it.
- Activity-bar TreeView (`src/ui/treeView.ts`) — registers `alEventLensView` as a `vscode.TreeView` populated from `EventIndexStore`. Publishers are grouped by source `appId` (`(workspace)` bucket first, then alphabetical), each leaf labeled `Kind::"Name" · EventName · (N)` where `N` is the live subscriber count from the store's resolver-keyed match. Trigger publishers participate alongside `[IntegrationEvent]` / `[BusinessEvent]`. Subscribes to `store.onDidChange` so the tree refreshes whenever `buildIndex` completes or the save watcher updates a file. Each leaf carries a `command: alEventLens.revealPublisher` payload — the panel PR will wire the handler.
- Subscriber-count CodeLens (`src/ui/codelens.ts`) — `registerCodeLens(context, store)` registers a `vscode.CodeLensProvider` for the `al` language that draws a `"N subscribers"` lens above each `[IntegrationEvent]` and `[BusinessEvent]` declaration in the active document, with live count computed from `EventIndexStore`. Pluralization: `1 subscriber` / `0 subscribers` / `N subscribers`. Clicking the lens fires `alEventLens.revealPublisher` with the publisher ref. Trigger publishers carry no source location and so get no lens. Gated by `alEventLens.codeLens.enabled`; the provider re-fires `onDidChangeCodeLenses` when the store mutates and when the setting toggles, so toggling takes effect without a reload. `extension.ts` now threads the store into the registration call.
- Mermaid export (`src/ui/mermaid.ts`, `src/index/match.ts`) — `renderMermaid(publisher, subscribers)` produces a deterministic `graph LR` diagram with one publisher node and `S1`..`SN` subscriber nodes sorted by `(owner.appId, owner.kind, owner.name, line)`; resolved subscribers use `-->`, unresolved use `-.->`; labels escape `"` → `&quot;` and newlines → `<br/>`. `alEventLens.exportMermaid` now reads the panel selection (or accepts an optional `Publisher` arg), copies the diagram to the clipboard via `vscode.env.clipboard`, and shows a confirmation toast. The panel now publishes its current selection back to the extension via a `selectionChanged` message, and `getSelectedPublisher()` is exported for command-palette invocations. The previously-duplicated subscriber-match logic in `resolver.ts`, `codelens.ts`, and `treeView.ts` is consolidated in the new `src/index/match.ts` (`publisherKey`, `subscriberKey`, `findSubscribersFor`, `countSubscribersByPublisherKey`); the webview keeps its own JS-encoded copy because it cannot import TS modules.
- Searchable webview panel (`src/ui/panel.ts`, `src/ui/panelHtml.ts`) — `openPanel(context, store)` creates a singleton `vscode.WebviewPanel` ("AL EventLens", `ViewColumn.Beside`) with a two-pane layout: a 30%-wide publisher list with case-insensitive substring search on the left, and a subscriber-detail pane on the right that lists each subscriber with `✓` / `⚠` resolved-status badges, file path, and line number. The panel subscribes to `EventIndexStore.onDidChange` so it re-paints on every workspace re-index and every incremental save. Self-contained HTML with a CSP-nonce-tagged inline script — no external resources, works identically on desktop and `vscode.dev`. The `alEventLens.refresh`, `alEventLens.revealPublisher`, and `alEventLens.gotoSubscriber` commands now have real implementations: refresh re-runs `buildIndex` into the store, revealPublisher opens the panel and posts a `select` message, and gotoSubscriber opens the target document at the subscriber's location. `extension.ts` constructs the store before registering the commands so the handlers can close over it.
- Resizable webview panel divider (`src/ui/panelHtml.ts`) — the previously-static 30%/70% split between the publisher list and the subscriber detail is now a 4-px draggable separator using `--vscode-panel-border` (tinted with `--vscode-focusBorder` on hover and during drag). Pointer events update a `--left-width` CSS variable, clamped to a 150-px minimum on the left and a 200-px minimum on the right. The chosen width is persisted via the webview's `vscode.setState` / `getState` so it survives panel hide/reveal (`retainContextWhenHidden` is already on); double-clicking the divider clears the override and snaps back to the 30% default.
- Per-row subscriber count in the webview panel's publisher list (`src/ui/panelHtml.ts`) — each list item now ends with ` · (N)` (the same format used by the tree leaves), where `N` is the live subscriber count computed via the existing webview-side `findSubscribersFor` helper. Resolved and unresolved subscribers are both included, consistent with the tree and the CodeLens.
- Hover tooltips on webview panel rows (`src/ui/panelHtml.ts`) — publisher rows show `"<Kind> event · N subscriber(s)\n<appId or (workspace)>"`; resolved subscriber rows show `"<kind>::\"<name>\"\n<full path>:<line>\nClick to jump to source"`; unresolved subscriber rows show the owner ref plus `"Unresolved — target app missing from .alpackages"`. Uses native HTML `title` attributes — no JS, no extra DOM, themed correctly inside the webview.
- Aggregate counts on TreeView intermediate rows (`src/ui/treeView.ts`) — `KindNode` and `ObjectNode` `TreeItem.description` fields show `(events / subscribers)` — the number of publisher events under that subtree plus the sum of subscriber counts across those events — so the tree can be skimmed without expanding everything and a busy subtree is obvious at a glance. `EventNode` leaves keep their existing `· (N)` subscriber count in the label; `AppNode` description continues to carry the `appPublisher` vendor name.
- Codicon icons on every TreeView node level (`src/ui/treeView.ts`) — `vscode.ThemeIcon` `iconPath` set on each row: `package` on app rows, a per-kind codicon on `KindNode` rows (`symbol-class` for codeunit, `symbol-struct` for table/tableextension, `window` for page/pageextension, `notebook` for report/reportextension, `search` for query, `file-code` for xmlport, `symbol-enum` for enum/enumextension, `lock` for permissionset, `symbol-interface` for interface), `symbol-file` on object rows, `symbol-event` on event leaves, and `info` on the empty-state placeholder. All icons tint with the current VS Code theme automatically — no extra assets in the bundle.

### Changed

- Dual-platform bundle: `package.json` now declares both `main` (Node bundle at `dist/extension.js`) and `browser` (Web Worker bundle at `dist/web/extension.js`). VS Code Desktop loads the Node bundle; `vscode.dev` and `github.dev` continue to load the browser bundle. Source-level web-discipline is preserved — the browser bundle is built from the same source and would fail to build if anything used Node-only APIs.
- Publishers tree (`src/ui/treeView.ts`) now shows the friendly app `Name` from each `.app`'s `NavxManifest.xml` as the row label, the `Publisher` (vendor) as the row description, and the full `appId` GUID in the tooltip. Buckets without a matching `appMeta` entry — older `.app` packages whose manifest omits `Name` — fall back to the raw GUID. Sort order is now case-insensitive on the displayed label (`(workspace)` still first). To support this, `parseManifest` in `src/symbols/appReader.ts` now extracts the optional `Name` / `Publisher` attributes (no throw when missing), `EventIndex` carries a `ReadonlyMap<string, AppMeta>` propagated through `EventIndexStore`, and the per-`.app` cache schema is bumped to v2 (`{ schemaVersion, publishers, name?, appPublisher? }`); pre-existing v1 cache entries are silently treated as misses and re-parsed. The new field is named `appPublisher` throughout — not `publisher` — to avoid colliding with AL EventLens's domain concept of an *event publisher*.
- TreeView hierarchy (`src/ui/treeView.ts`) now drills four levels — source app → AL object kind → object → event — instead of one flat publisher row per event under the app bucket. `KindNode` rows (`Codeunit`, `Table`, `XmlPort`, ...) sort alphabetically by display label; `ObjectNode` rows sort case-insensitively by `owner.name`; `EventNode` leaves sort by event name and keep the existing `· (N)` subscriber count plus the `alEventLens.revealPublisher` click command. Trigger publishers participate alongside `[IntegrationEvent]` / `[BusinessEvent]` publishers under their respective kind bucket. The leaf-label format changed from `Kind::"Name" · EventName · (N)` to `EventName · (N)` since `Kind` and `Name` now live on the parent rows.

### Fixed

- `.alpackages/*.app` publishers inside `Namespaces[]` were silently dropped, so most of Microsoft BaseApp / System Application — including `Codeunit::"Sales-Post"` and everything else under a BC 24+ namespace — never appeared in the panel, tree, or CodeLens counts (`src/symbols/detect.ts`, `src/index/cache.ts`). The schema-dispatch helper in `detect.ts` gated nested-schema routing on `typeof json === 'object'`, but the indexer at `src/index/indexer.ts:86` passes `AppContents.symbolReferenceJson` — a string — so the check always failed and every package was force-routed to the flat-only walk. Worse, BC 26+ Microsoft packages carry **both** a small top-level `Codeunits[]` (legacy un-namespaced holdovers) AND a large `Namespaces[]` tree in the same file, so even an already-parsed object would have lost the nested side; the schemas are additive, not mutually exclusive. Real-world impact on a stock BC 28 .alpackages folder: BaseApp went from 8 extracted publishers to 23,498. `parseSymbolReference` now unconditionally routes through the unified namespace walker (which extracts from the root container and recurses into any `Namespaces[]` entries), correctly handling both schemas and any hybrid. Cache schema bumps to v4 to invalidate previously-cached poisoned entries from the broken dispatcher. New regression tests cover the two cases the previous dispatch suite missed: a JSON string carrying the nested schema, and a single SymbolReference with both top-level arrays AND `Namespaces[]`.
- Activity bar icon (`package.json`, `images/activity-bar.svg`) was rendering as a grey placeholder because the contributed `viewsContainers.activitybar` entry pointed at `images/icon.png`, and VS Code's activity bar refuses to render colored PNGs — it requires a monochrome SVG that uses `currentColor` so it can re-tint per theme. Added a separate `images/activity-bar.svg` (magnifying glass + 3 connected event nodes, matching the marketplace PNG's concept in monochrome) and pointed the activity-bar `icon` field at it; the top-level marketplace `icon` field still uses `images/icon.png`.
- Workspace events would not appear in the tree if you launched VS Code into an empty workspace and then opened a project folder (`src/index/folderWatcher.ts`, `src/extension.ts`). The extension only ran `buildIndex` at activation time — `onDidChangeWorkspaceFolders` was not wired up, so folders added after launch left the store untouched until the user manually invoked `AL EventLens: Refresh Index`. New `registerWorkspaceFolderReindex` listens for folder add/remove and re-runs the indexer, piping the result into the store the same way the activation path does. The shared `runIndexWithProgress` helper (`src/index/reindex.ts`) factors out the progress-wrapped `buildIndex` call so the activation path, the `alEventLens.refresh` command, and the folder watcher all share one implementation. Test coverage: 4 new `folderWatcher` tests; 6 new end-to-end `alEventLens.gotoSubscriber` command tests covering every shape that arrives at the command (`{_start, _end}`, `{start, end}`, real `vscode.Location`, `al-eventlens-app:` short-circuit, no-arg palette path, degenerate empty range); 2 new indexer tests asserting workspace publishers (and synthesized workspace trigger publishers) carry `owner.appId === undefined`; 1 new TreeView test that drills the full App → Kind → Object → Event path under the `(workspace)` bucket as a regression for "workspace events not listed".
- Clicking an unresolved subscriber row in the panel detail pane did nothing (`src/ui/panelHtml.ts`). The jump-to-source click handler was nested inside the `if (s.resolved)` branch, so any subscriber whose target publisher couldn't be found — typically because the target app is missing from `.alpackages`, or there's a typo in the subscription — had no click handler at all, even though the subscriber's OWN source location (its `[EventSubscriber]` attribute in the user's workspace) is always valid. The `resolved` flag is about the target publisher, not the subscriber's source, so the click handler now attaches in both cases. Unresolved rows keep their `⚠` badge and `target app missing from .alpackages` trail (useful context) but clicking jumps to the subscriber's source so the user can inspect or fix the subscription. The `cursor: default` CSS override on `.sub-list li.unresolved` is dropped to match.
- Tree-click filter syntax (`src/ui/panelHtml.ts`, `src/ui/panel.ts`) was emitting the verbose `app:<guid> kind:<k> object:"<name>"` form — the `<guid>` polluted the search box and the app filter unnecessarily narrowed away same-named objects from sibling apps. Tree clicks now emit an AL-style identity selector instead: `Codeunit::"Sales-Post"` (or unquoted when the name has no spaces). The panel's search box gains a `<kind>::<name>` / `<kind>::"name with spaces"` token type recognized by `parseSearch`, exact-matched in `passesFilter` (case-insensitive), and re-emitted by `buildSearchText`. The Kind dropdown still reflects the selector's kind; changing the dropdown clears the identity selector so the two filter sources don't contradict. The existing `app:` / `kind:` / `object:` prefixes still work for typed input and dropdowns.
- Clicking a subscriber row in the panel detail pane crashed the `alEventLens.gotoSubscriber` command with `TypeError: Cannot read properties of undefined (reading 'line')` and silently did nothing (`src/extension.ts`, `src/util/reviveLocation.ts`). `vscode.Range` exposes its `start` / `end` as class getters with data stored in internal `_start` / `_end` slots; the webview `postMessage` serializer doesn't copy class getters, so a `vscode.Range` round-tripped through a webview arrives on the host as `{_start, _end}` instead of `{start, end}`. The command was reading `loc.range.end.line` directly and crashing. New `reviveRange` helper tolerates `{start, end}`, `{_start, _end}`, a degenerate empty object (falls back to a `(0, 0)` Range so the file still opens), and a real `vscode.Range` instance from in-process callers. The existing panel test exercised the dispatch path with a real `vscode.Location` and so missed the cloned-shape regression; a new dispatch test passes the `{_start, _end}` shape that production actually produces.
- Tree view showed `"No publishers indexed yet — try AL EventLens: Refresh Index"` on initial load even while the status bar was still reporting indexing progress (`src/ui/treeView.ts`, `src/index/store.ts`, `src/extension.ts`). The store couldn't distinguish "indexing hasn't finished yet" from "indexed and empty" — both yielded a zero-length publisher list. New `isInitialized` flag on `EventIndexStore` flips on the first `set` / `updateFile`; the tree renders an `"Indexing workspace…"` row with a spinning `sync~spin` codicon until then, then transitions to the real empty-state or populated tree on the first `onDidChange` fire. The activation-time `indexWithProgress().catch` now also calls `store.set({...empty})` so a failed initial scan progresses the tree past the spinner instead of leaving it stuck.
- Duplicate publishers from `.alpackages/*.app` packages that ship bundled `src/*.al` source (`src/index/indexer.ts`) — every event under any such package (Microsoft BaseApp, Business Foundation, Test Runner, etc.) was appearing twice in the panel list, tree, and CodeLens counts. The indexer was pushing publishers from both `parseSymbolReference` (the authoritative source per CLAUDE.md) AND from `parseAl` over the bundled `src/*.al` files. The bundled-source pass now contributes subscribers and trigger owners only; publishers come exclusively from `SymbolReference.json`. New regression test asserts an event declared in both surfaces appears exactly once.
